# 共同編集の手順

このファイルは、**共同でコードを編集する人**（エンジニアでない方も）向けの手順です。  
「初回だけやること」と「2回目から毎回やること」の2つに分かれています。

---

## 用語を1行で

- **ターミナル** … Cursor や VS Code の下にある「黒い画面」。ここにコマンド（英数字の命令）を打ちます。
- **プロジェクトのフォルダ** … このソフトの入っているフォルダ（例: `web-order-co`）。ターミナルで「このフォルダの中にいる」状態でコマンドを実行します。
- **pull** … インターネット上（GitHub）にある「最新版」を、自分の PC に取り込むこと。
- **push** … 自分が編集した内容を、インターネット上（GitHub）に送ること。

---

## あなた（リポジトリの持ち主）がやること

1. **招待**  
   GitHub でリポジトリを開く → **Settings** → **Collaborators** → **Add people** → 相手のユーザー名かメールを入力 → **Write** で追加。相手に招待メールが届きます。

2. **手順の共有**  
   相手に「このファイル（共同編集の手順.md）の『初回の流れ』と『2回目からの流れ』を見て作業してください」と伝える。

---

# 共同編集する人向け

## 初回の流れ（最初の1回だけ）

**目的**: 自分の PC に「このプロジェクト」を用意して、Cursor で開けるようにする。

| 順 | どこで | やること |
|----|--------|----------|
| 1 | **メール** | 届いた招待メールの **Accept invitation** をクリックする。 |
| 2 | **ターミナル**（Cursor の下の黒い画面。場所はどこでもよい） | 次の3行を、1行ずつ順番に打って Enter する。<br>① `git clone https://github.com/hfmmmi/web-order-co.git`<br>　→ プロジェクトを自分の PC にコピーします。<br>② `cd web-order-co`<br>　→ そのフォルダの中に入ります。<br>③ `npm install`<br>　→ 必要なライブラリを入れます。 |
| 3 | **Cursor** | メニュー **File** → **Open Folder** を選び、今コピーした **web-order-co** フォルダを選んで開く。 |

ここまでで「初回の準備」は終わりです。次からは「2回目からの流れ」だけやれば大丈夫です。

---

## 2回目からの流れ（毎回この順番で）

**目的**: 最新の状態で編集して、編集した内容を GitHub に送る。

流れは **pull → 編集 → テスト → push** の4つです。

### ① pull（最新版を取ってくる）

**なぜ**: ほかの人が先に変更を送っているかもしれないので、作業を始める前に「いちばん新しい版」を自分の PC に取り込みます。Cursor は自動で取り込まないので、**必ず手動でやる**必要があります。

**どこで**: ターミナル（プロジェクトのフォルダを Cursor で開いた状態で、そのフォルダが開いているターミナルを使う）。

**やること**:
1. プロジェクトのフォルダにいるか確認（いなければ `cd web-order-co` のようにフォルダ名を打って入る）。
2. 次のコマンドを打って Enter: **`git pull origin main`**


---

### ② 編集（ファイルを直す）

**どこで**: Cursor で、必要なファイルを開いて編集する。いつも通りでOK。

---

### ③ テスト（動きが壊れていないか確認する・推奨）

**なぜ**: 編集したせいで動かなくなっていると、そのまま送るとみんなに迷惑がかかるので、送る前に「テスト」で確認することをおすすめします。所要時間10～20分

**どこで**: ターミナル（プロジェクトのフォルダで）。

**やること**: 次のコマンドを打って Enter。  
**`$env:DATA_DIR = "tests/_sandbox_data"; npm run test:all`**  
エラーが出ずに終わればOK。問題があれば直してから次へ。

---

### ④ push（編集した内容を GitHub に送る）

**どこで**: ターミナル（プロジェクトのフォルダで）。

**やること**: 次の4つを、**順番に**1つずつ打って Enterする。

| 順 | 打つコマンド | 意味（なんとなく） |
|----|--------------|---------------------|
| 1 | `cd web-order-co`（まだフォルダに入っていない場合） | プロジェクトのフォルダに入る |
| 2 | `git add .` | 「今の変更を全部、送る候補に入れる」 |
| 3 | `git commit -m "○○の修正"` | 「○○の修正」の部分を、今回やった内容に書き換える（例: ログイン画面の文言を直した → `"ログイン画面の文言を修正"`） |
| 4 | `git push origin main` | インターネット上（GitHub）に送る |

※ パスワードやログインを聞かれたら、GitHub のユーザー名とパスワード（または Personal Access Token）を入力する。

※ 送れたかどうかは、GitHub のリポジトリのページを開くと、今の内容が更新されて表示されます。

---

## 共同編集者に渡す一文（コピペ用）

> 共同編集の招待を送りました。メールの **Accept** を押したら、リポジトリにある **docs/共同編集の手順.md** を開いてください。「初回の流れ」を最初の1回だけやり、「2回目からの流れ」を毎回（pull → 編集 → テスト → push）やってください。

---

## 補足（必要なら読む）

- **（任意）テスト必須ルール**  
  「GitHub 上のテストが通っていないと、main に取り込めない」ようにしたい場合は、GitHub の **Settings** → **Branches** → **Add rule**（ブランチ名 `main`）→ **Require status checks** で **test** にチェック → 保存。くわしくは `docs/ci-setup.md`。
- **本番** … ここでは「GitHub の main」＝みんなが参照する最新版のことを指します。`git push origin main` を実行した時点で、その内容が GitHub の main に保存されます。
- **本番サーバ（VPS）にプログラムを反映する** … GitHub の `main` に push したあと、**VPS に SSH して** `git pull` とアプリの再起動を行います。手順は **`docs/vps-production-runbook.md` の「11. 本番デプロイ」**（その文書の冒頭「読み方」も参照）を見てください。
