# 共同編集の手順

このファイルは、**共同でコードを編集する人**（エンジニアでない方も）向けの手順です。  
「初回だけやること」と「2回目から毎回やること」の2つに分かれています。

---

## 用語を1行で

- **ターミナル** … Cursor や VS Code の下にある「黒い画面」。ここにコマンド（英数字の命令）を打ちます。
- **プロジェクトのフォルダ** … このソフトの入っているフォルダ（例: `web-order-co`）。ターミナルで「このフォルダの中にいる」状態でコマンドを実行します。
- **pull** … インターネット上（GitHub）にある「最新版」を、自分の PC に取り込むこと。
- **push** … 自分が編集した内容を、インターネット上（GitHub）に送ること。

---

## あなた（リポジトリの持ち主）がやること

1. **招待**  
   GitHub でリポジトリを開く → **Settings** → **Collaborators** → **Add people** → 相手のユーザー名かメールを入力 → **Write** で追加。相手に招待メールが届きます。

2. **手順の共有**  
   相手に「このファイル（共同編集の手順.md）の『初回の流れ』と『2回目からの流れ』を見て作業してください」と伝える。

---

# 共同編集する人向け

## 初回の流れ（最初の1回だけ）

**目的**: 自分の PC に「このプロジェクト」を用意して、Cursor で開けるようにする。

| 順 | どこで | やること |
|----|--------|----------|
| 1 | **メール** | 届いた招待メールの **Accept invitation** をクリックする。 |
| 2 | **ターミナル**（Cursor の下の黒い画面。場所はどこでもよい） | 次の3行を、1行ずつ順番に打って Enter する。<br>① `git clone https://github.com/hfmmmi/web-order-co.git`<br>　→ プロジェクトを自分の PC にコピーします。<br>② `cd web-order-co`<br>　→ そのフォルダの中に入ります。<br>③ `npm install`<br>　→ 必要なライブラリを入れます。 |
| 3 | **Cursor** | メニュー **File** → **Open Folder** を選び、今コピーした **web-order-co** フォルダを選んで開く。 |

ここまでで「初回の準備」は終わりです。次からは「2回目からの流れ」だけやれば大丈夫です。

---

## 2回目からの流れ（毎回この順番で）

**目的**: 最新の状態で編集して、編集した内容を GitHub に送る。

流れは **pull → 編集 → テスト → push** の4つです。

### ① pull（最新版を取ってくる）

**なぜ**: ほかの人が先に変更を送っているかもしれないので、作業を始める前に「いちばん新しい版」を自分の PC に取り込みます。Cursor は自動で取り込まないので、**必ず手動でやる**必要があります。

**どこで**: ターミナル（プロジェクトのフォルダを Cursor で開いた状態で、そのフォルダが開いているターミナルを使う）。

**やること**:
1. プロジェクトのフォルダにいるか確認（いなければ `cd web-order-co` のようにフォルダ名を打って入る）。
2. 次のコマンドを打って Enter: **`git pull origin main`**


---

### ② 編集（ファイルを直す）

**どこで**: Cursor で、必要なファイルを開いて編集する。いつも通りでOK。

---

### ③ テスト（動きが壊れていないか確認する・推奨）

**なぜ**: 編集したせいで動かなくなっていると、そのまま送るとみんなに迷惑がかかるので、送る前に「テスト」で確認することをおすすめします。所要時間10～20分

**どこで**: ターミナル（プロジェクトのフォルダで）。

**やること**: 次のコマンドを打って Enter。  
**`$env:DATA_DIR = "tests/_sandbox_data"; npm run test:all`**  
エラーが出ずに終わればOK。問題があれば直してから次へ。

---

### ④ push（編集した内容を GitHub に送る）

**どこで**: ターミナル（プロジェクトのフォルダで）。

**やること**: 次の4つを、**順番に**1つずつ打って Enterする。

| 順 | 打つコマンド | 意味（なんとなく） |
|----|--------------|---------------------|
| 1 | `cd web-order-co`（まだフォルダに入っていない場合） | プロジェクトのフォルダに入る |
| 2 | `git add .` | 「今の変更を全部、送る候補に入れる」 |
| 3 | `git commit -m "○○の修正"` | 「○○の修正」の部分を、今回やった内容に書き換える（例: ログイン画面の文言を直した → `"ログイン画面の文言を修正"`） |
| 4 | `git push origin main` | インターネット上（GitHub）に送る |

※ パスワードやログインを聞かれたら、GitHub のユーザー名とパスワード（または Personal Access Token）を入力する。

※ 送れたかどうかは、GitHub のリポジトリのページを開くと、今の内容が更新されて表示されます。

※ **本番稼働（SaaS 開始）後**は、業務 JSON を commit しないでください。くわしくは下記 **「鉄のルール」** を参照。

---

## 共同編集者に渡す一文（コピペ用）

> 共同編集の招待を送りました。メールの **Accept** を押したら、リポジトリにある **docs/共同編集の手順.md** を開いてください。「初回の流れ」を最初の1回だけやり、「2回目からの流れ」を毎回（pull → 編集 → テスト → push）やってください。**本番稼働後は「鉄のルール」— 開発側から本番の業務データを一切触れない — を必ず守ってください。**

---

## 補足（必要なら読む）

- **（任意）テスト必須ルール**  
  「GitHub 上のテストが通っていないと、main に取り込めない」ようにしたい場合は、GitHub の **Settings** → **Branches** → **Add rule**（ブランチ名 `main`）→ **Require status checks** で **test** にチェック → 保存。くわしくは `docs/ci-setup.md`。
- **本番** … ここでは「GitHub の main」＝みんなが参照する最新版のことを指します。`git push origin main` を実行した時点で、その内容が GitHub の main に保存されます。
- **本番サーバ（VPS）への反映** … 下記 **「本番（VPS）への反映手順」** を参照。くわしい VPS 操作は **`docs/vps-production-runbook.md`** も併せて読んでください。

# 本番（VPS）への反映手順

**対象**: リポジトリの持ち主・本番反映の担当者。  
**本番 URL の例**: `https://order.hifumi-m.com`  
**VPS の例**: `162.43.47.219`（IP はパネルで確認）

本番には **2種類のもの** があります。混同しないでください。

| 種類 | 中身の例 | 反映のしかた |
|------|----------|--------------|
| **プログラム** | HTML・CSS・JavaScript・サーバー処理（`public/` など） | **GitHub 経由**（`git push` → VPS で `git pull`） |
| **業務データ** | 顧客・商品・注文・設定（`*.json`） | **scp でファイルコピー**（Git とは別） |

**HTML や画面デザインはプログラム側** です。JSON を上げただけではデザインは変わりません。  
**データを上げただけではコードは変わりません。** 両方必要なときは **両方の手順** を行います。

---

## 鉄のルール（SaaS / 本番稼働後）

> **開発側（ローカル PC・GitHub・共同編集者の作業環境）から、本番の業務データを一切触れてはいけない。**

**本番稼働（SaaS 開始）以降、例外なし。** リポジトリの持ち主・共同編集者・デプロイ担当者、全員が守ります。

### 業務データとは

本番 VPS 上の **顧客・注文・商品マスタの中身・価格・設定など、業務で使われる JSON および添付ファイル** のことです。  
（プログラム本体 `public/`・`routes/`・`services/` 等は **対象外** — これらは Git で更新します。）

### 開発側からしてはいけないこと

| 禁止 | 例 |
|------|-----|
| 本番の JSON を **上書き** | ローカルの `customers.json` / `orders.json` 等を **scp で本番へ送る** |
| 本番データを **Git 経由で上書き** | 業務 JSON を commit & push し、本番で `git pull` して反映する |
| 本番サーバで **テストを実行** | VPS 上で `npm test` / `npm run test:all` |
| 本番データを **無バックアップで直接編集** | SSH で本番の JSON を手編集（復旧不能になる） |

### 開発側からしてよいこと

| 許可 | 例 |
|------|-----|
| **プログラムだけ** 本番へ反映 | パターンA（`git push` → 本番 `git pull` → restart） |
| 本番データを **読む・コピーする**（本番 → 開発） | scp で本番 → ローカルへ取り込み、**ローカルでのみ** 参照・テスト |
| ローカル・CI では **サンドボックスのみ** | `DATA_DIR=tests/_sandbox_data` 付きでテスト |

### 本番の業務データを変える必要があるとき

**本番の管理画面・本番 URL 上の操作**、または **販管連携 API** など、**本番環境の正当な経路** で行います。  
開発 PC から本番ファイルを直接書き換える方法は使いません。

### commit 時の注意（共同編集者向け）

本番稼働後は、業務 JSON（`customers.json` / `orders.json` / `products.json` 等）を **GitHub に送らない** ようにします。  
`git add .` の代わりに、**変更したコードのフォルダだけ** を add してください（例: `git add public/ middlewares/`）。

---

## データの「正」（どちらを優先するか）

**自動で双方向マージする機能は現状ありません。** 上書きする前に、どちらを正とするか決めてください。

| データ | 開発中（本番未稼働） | **本番稼働後**（鉄のルール） |
|--------|----------------------|----------------------------|
| 商品・ランク価格・個別価格 | ローカル → 本番（scp 可） | **本番が正**。開発から上書き禁止。**本番管理画面** で更新 |
| 顧客マスタ | ローカル → 本番（scp 可） | **本番が正**。開発から上書き禁止。**本番管理画面** で更新 |
| システム設定 `settings.json` | ローカル → 本番 | **本番が正**。開発から上書き禁止。文面等は **本番管理画面** で更新（SMTP パスワードは本番 `.env`） |
| **注文 `orders.json`** | ローカル → 本番 | **本番が正**。開発から触れない。必要なら **本番 → ローカル** に取り込むのみ |
| サポート `support_tickets.json` | ローカル → 本番 | **本番が正**。開発から触れない |
| 買取依頼 `kaitori_requests.json` | ローカル → 本番 | **本番が正**。開発から触れない |
| 管理者 `admins.json` | ローカル → 本番 | **本番が正**。開発から上書き禁止 |

### 注文履歴などが本番にたまっている場合は？

**ローカルの `orders.json` をそのまま本番に scp すると、本番の注文履歴はローカル版で上書きされ、本番だけにあった注文は消えます。**

本番稼働後の基本方針:

1. **鉄のルール** … 開発側から本番の業務データを **一切触れない**（上書き・scp 送信・Git 経由の反映すべて禁止）。
2. **注文・サポート・買取など「起きた出来事」のデータ** … **本番に残す**。開発するときは **本番 → ローカル** へ取り込んで参照する。
3. **商品・価格・画面・機能** … **プログラム** は GitHub → 本番（パターンA）。**マスタの中身** は本番管理画面で更新する。
4. **顧客マスタ** … 本番でログイン・登録が始まったら、開発側からの上書きは **禁止**。

**本番 → ローカルへ注文を取り込む例**（PowerShell・プロジェクトフォルダで）:

```powershell
cd C:\Users\Owner\Desktop\program\web-order
$KEY = "$env:USERPROFILE\.ssh\id_ed25519"
scp -i $KEY root@162.43.47.219:/opt/weborder/orders.json .\orders.json
scp -i $KEY root@162.43.47.219:/opt/weborder/support_tickets.json .\support_tickets.json
```

※ 取り込んだあと、ローカルで `npm run dev` すると **本番と同じ注文** が見えます。  
※ 将来、注文の自動マージや販管連携での同期を強化する場合は、別途設計・実装が必要です（現状は **手順で運用**）。

## 事前準備（初回だけ）

1. **SSH 鍵** … 旧 PC の `C:\Users\（自分）\.ssh\id_ed25519` を新 PC にコピーする。
2. **接続確認**（PowerShell）:

   ```powershell
   ssh -i $env:USERPROFILE\.ssh\id_ed25519 root@162.43.47.219
   ```

   `root@...#` と出れば成功。終わったら `exit`。

3. **本番のデータ場所**（VPS 上）… 多くの場合 **`/opt/weborder`**（`DATA_DIR` 未設定時）。  
   確認: `grep DATA_DIR /opt/weborder/.env`（未設定なら `/opt/weborder` 直下の JSON）

## パターンA: プログラムだけ更新（通常のデプロイ）

**画面・機能の変更を本番に出すとき**（データは触らない）。  
**2回目からの流れ** の push まで終わっている前提。

### ① パソコン（PowerShell）

```powershell
cd C:\Users\Owner\Desktop\program\web-order
git pull origin main
$env:DATA_DIR = "tests/_sandbox_data"; npm run test:api
git add .
git commit -m "変更内容の説明"
git push origin main
```

### ② VPS（SSH・`root@...#`）

```bash
cd /opt/weborder
git pull origin main
npm ci --omit=dev
systemctl restart weborder
git log -1 --oneline
curl -sS -o /dev/null -w "%{http_code}\n" http://127.0.0.1:3000/
```

`git pull` で JSON の競合が出た場合は **パターンB の「git pull が JSON で止まるとき」** へ。

### ③ ブラウザ

`https://order.hifumi-m.com` を **Ctrl + Shift + R** で強制再読み込み。

## パターンB: プログラム + 業務データを反映（本番稼働前のみ）

> **本番稼働（SaaS 開始）後は、鉄のルールによりパターンBは使用禁止。** マスタ投入が必要な場合も、本番管理画面または別途合意した本番側の手順で行う。

**ローカルの JSON を本番に載せたいとき**（初回投入・本番未稼働時のマスタ一括更新）。  
**必ず本番バックアップを取ってから** 実行してください。

### 反映前チェックリスト

- [ ] 本番に **新しい注文** が入っていないか確認した（入っているなら `orders.json` は上げない）
- [ ] ローカルの JSON が **意図した最新** である
- [ ] 反映時間帯を決めた（`weborder` 停止中はサイトが使えない）
- [ ] `$env:DATA_DIR` を **付けずに** ローカル開発していた（テスト用データと混同していない）

### ① VPS: 停止・バックアップ

```bash
cd /opt/weborder
systemctl stop weborder

mkdir -p /root/prod-json-backup
cp -a orders.json products.json rank_prices.json rank_prices_updated_at.json \
  support_tickets.json customers.json settings.json admins.json prices.json \
  stocks.json kaitori_master.json kaitori_requests.json estimates.json \
  /root/prod-json-backup/ 2>/dev/null

tar czf /root/weborder-data-backup-$(date +%Y%m%d-%H%M).tar.gz -C /opt/weborder \
  customers.json products.json prices.json rank_prices.json \
  rank_prices_updated_at.json admins.json orders.json \
  stocks.json kaitori_master.json kaitori_requests.json \
  support_tickets.json estimates.json
```

※ `settings.json` が無い環境では tar の対象から外してよい。

### ② パソコン: JSON を scp（PowerShell・別ターミナル）

**上げるファイルは目的に応じて選ぶ。** 本番稼働後は **`orders.json` を含めない** のが安全なことが多い。

```powershell
cd C:\Users\Owner\Desktop\program\web-order
$KEY = "$env:USERPROFILE\.ssh\id_ed25519"
$REMOTE = "root@162.43.47.219"
$DEST = "/opt/weborder"

# マスタ中心（注文履歴は本番を残す例）
$files = @(
  "customers.json","products.json","prices.json","rank_prices.json",
  "rank_prices_updated_at.json","settings.json","admins.json",
  "stocks.json","kaitori_master.json","kaitori_requests.json",
  "support_tickets.json","estimates.json"
)
# 注文も含めてローカルで全上書きする場合だけ orders.json を追加（本番未稼働時など）

scp -i $KEY @files "${REMOTE}:${DEST}/"
```

### ③ パソコン: プログラムを push（まだの場合）

パターンA の ① と同じ（`git push origin main`）。

### ④ VPS: コード取得（JSON 競合対策つき）

```bash
cd /opt/weborder

# いまの JSON を再度退避（scp 後の状態）
cp -a orders.json products.json rank_prices.json rank_prices_updated_at.json \
  support_tickets.json customers.json settings.json admins.json prices.json \
  stocks.json kaitori_master.json kaitori_requests.json estimates.json \
  /root/prod-json-backup/ 2>/dev/null

# Git が管理している JSON だけ一旦リセット（pull を通すため）
git restore orders.json products.json rank_prices.json rank_prices_updated_at.json support_tickets.json

git pull origin main
npm ci --omit=dev

# scp / 退避した業務データを戻す
cp -a /root/prod-json-backup/*.json /opt/weborder/

systemctl start weborder
systemctl is-active nginx weborder
curl -sS -o /dev/null -w "%{http_code}\n" http://127.0.0.1:3000/
git log -1 --oneline
```

成功の目安:

- `git log -1` が push したコミット（例: `92aa256 システム設定変更`）
- `curl` が `200`
- ブラウザで新 UI が見える

### git pull が JSON で止まるとき

```
error: Your local changes to the following files would be overwritten by merge:
        orders.json ...
Aborting
```

→ 上記 **④** のとおり、`/root/prod-json-backup/` へ退避 → `git restore`（対象 JSON のみ）→ `git pull` → `cp` で戻す。

---

## やってはいけないこと

| 禁止 | 理由 |
|------|------|
| **本番稼働後、開発側から本番の業務 JSON を触る** | **鉄のルール**（scp 上書き・Git 経由の反映・本番での手編集すべて） |
| VPS 上で `npm test` / `npm run test:all` | 本番 JSON を壊す恐れ |
| 本番稼働中に `orders.json` を安易にローカルから上書き | **本番の注文履歴が消える** |
| 業務 JSON を `git add .` で GitHub に push | 本番 `git pull` で上書きされる恐れ |
| PowerShell の scp を **SSH 画面（`root@...#`）** で実行 | パスが通らず失敗する |
| `git stash` に **Git 管理外の `settings.json`** を含める | stash 全体が失敗する |
| バックアップなしのデータ上書き | 復旧不能になる |

---

## 復旧（問題が起きたとき）

VPS 上:

```bash
cd /opt/weborder
systemctl stop weborder
tar xzf /root/weborder-data-backup-（日時）.tar.gz -C /opt/weborder
# または
cp -a /root/prod-json-backup/*.json /opt/weborder/
systemctl start weborder
```
