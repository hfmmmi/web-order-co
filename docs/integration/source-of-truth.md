# データの正（Source of Truth）と販管連携の前提

WEB 受注（web-order）と販売管理（sales-mgmt）の双方向連携で、**同一レコードを両方が「勝手に正」として書き換えない**ための初期方針です。運用で変える場合は本書を更新してください。

## 確定した前提（計画フェーズより）

- 販管の第1段階の永続化: **JSON ファイル**（`DATA_DIR` 相当のパターンで web-order と揃える）
- 連携の志向: **双方向**（ただし「正」はエンティティごとに一方）

## エンティティごとの「正」の推奨（初期案）

| ドメイン | 正（マスタ／発生源） | もう一方の扱い |
|----------|----------------------|----------------|
| 受注トランザクション（注文の発生） | **WEB 受注** | 販管はミラー取込・売上・請求のため参照。販管からの注文「新規作成」は別フローで設計するまで行わない |
| 注文の履行（出荷・追跡番号・ステータス） | **WEB 受注**（現行 UI・CSV）を正としつつ、将来は販管からの更新を **連携 API のみ** で受ける | 販管→WEB の書き戻しは専用エンドポイント＋キューで直列化 |
| 顧客マスタ（ID・ログイン） | **WEB 受注**（顧客がログインするため） | 販管は請求先・与信など **WEB に無い属性** を販管側 JSON で保持。WEB 側に持たせる属性の更新は **連携パッチ API のみ**（パスワードは連携では変更しない） |
| 商品マスタ・表示用価格 | **WEB 受注** | 販管の原価・仕入は販管のみ。WEB の売価を販管が上書きする場合は明示フェーズで API を拡張 |
| ランク価格・特別価格 | **WEB 受注** | 販管は参照用コピーまたは都度取得 |
| 在庫 | **WEB 受注**（現行の stock 系）を正とする運用を維持 | 販管で在庫を持ち始める場合は「同期方向」をドキュメント化してから |

## 販管 → WEB 第1弾の書き戻し範囲（実装済みの起点）

第1弾として、次のみを連携で更新対象とする（[integration-api ルート](../../routes/integration-api.js)）。

- **顧客の表示・価格関連**: `customerName`, `email`, `priceRank`（いずれも任意フィールド・部分更新可）
- **パスワード**: 連携では **変更不可**（セキュリティ・監査のため）
- **冪等**: リクエストに `idempotencyKey` を付与可能。同一顧客に同一キーの再送は成功扱いでノーオペレーション

注文・商品の販管→WEB 書き戻しは、契約（OpenAPI）拡張とキュー設計が済んでから別タスクとする。

## 環境変数（WEB 受注）

| 変数 | 説明 |
|------|------|
| `ERP_SYNC_API_KEY` | 販管などバックエンド連携用の共有秘密。未設定時は連携ルートは 503 で無効 |

ローカルでマスタが空のとき、販管からの `GET /v1/products` / `v1/customers` の検証用に **web-order** で `npm run seed:erp-dev` を実行できます（空ファイルのときだけサンプルを投入。本番 `NODE_ENV=production` では中止）。

## 連携エンドポイント（WEB 受注）

ベースパス: `/api/integration`

| メソッド | パス | 概要 |
|----------|------|------|
| GET | `/v1/orders` | 注文スナップショット（`since`, `limit`） |
| GET | `/v1/customers` | 顧客マスタスナップショット（`limit`・パスワードなし・販管へのコピー用） |
| GET | `/v1/products` | 商品マスタスナップショット（`limit`, `activeOnly`・ランク別単価の詳細は含まない） |
| POST | `/v1/customers/patch` | 顧客の部分更新（パスワード不可・冪等キー可） |
