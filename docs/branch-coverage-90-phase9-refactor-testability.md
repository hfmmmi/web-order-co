# 分岐カバレッジ 90% — フェーズ9（テスト容易性のための最小リファクタ）

カバレッジ不足を **プロダクションの大規模書き換え**で解消しようとすると、レビュー負荷とバグリスクが跳ね上がる。フェーズ9では **テストを書きやすくするための最小差分**だけを許容する。

## やるべきときのサイン

- **同じ分岐**を API テストで取るのに、**長いセットアップ**や **複数エンドポイントの順番依存**が毎回必要になっている。
- **純粋な計算・パース**が巨大なハンドラ内にあり、ユニットテストで切り出せそうな塊がある。
- **モック対象が多すぎて**テストが壊れやすい（1 モジュールに副作用が集中）。

## 許容しやすいリファクタ（例）

| 種類 | 例 | 注意 |
|------|-----|------|
| **純粋関数の抽出** | 日付パース・CSV 行の正規化を `utils/` または `services/` 内の関数に分離 | 振る舞いは変えず、既存テストが通ること |
| **依存の引数化** | `fs` やパスを引数で渡せるようにし、テストでスタブを差し込む | 公開 API の破壊的変更は避ける |
| **重複 `if` の統合** | 同じ条件が複数箇所にあるときの共通化 | 仕様変更に見えないよう PR 説明を明確に |

## やらないこと（カバレッジのためだけに避ける）

- **仕様のない「将来のための」抽象化**や、広範な **ディレクトリ再編**。
- **挙動が変わるリファクタ**をカバレッジ PR に混ぜる（別 PR に分ける）。
- **テスト専用のフック**を本番コードに大量追加する（`if (process.env.NODE_ENV === 'test')` の乱立など）。

## 判断の目安

1. **テスト追加だけで分岐が埋まる**なら、先にテスト（フェーズ3〜7）。
2. どうしても **セットアップが重すぎる**と判断したら、**抽出する関数の入出力**を文章化してから最小パッチにする。
3. PR は **「リファクタ」と「テスト」**でコミットを分けるか、説明で境界を明示する。

## フェーズ9の完了条件

1. 本ドキュメントの **許容範囲と禁止事項**がチームで共有できていること。
2. リファクタ PR に **テスト追加または既存テストの緑**がセットになっていること。
